Техническое задание Backend (v2)
Платформа онлайн-бронирования столиков «KeepBooking»

Документ является переработанной и расширенной версией tz.txt.
Сохранена вся исходная функциональность, добавлены разделы, необходимые
для системы production-уровня (доступность столиков, конкурентность,
идемпотентность, обработка ошибок, тестирование, CI/CD, миграции и др.).

================================================================
0. Что изменилось относительно tz.txt (сводка)
================================================================

Добавлено / уточнено:
  - Раздел «Доступность столиков и слоты бронирования» (главный пробел tz.txt).
  - Гарантии конкурентности при бронировании (защита от double-booking).
  - Идемпотентность операций записи (Idempotency-Key).

  - Хранение файлов (S3/MinIO) вместо «url в базе» без уточнений.
  - Асинхронные уведомления через outbox-паттерн + брокер.
  - Кэширование (Redis): что именно и с какими TTL.
  - Стратегия версионирования API и обратной совместимости.
  - Требования к тестам (unit / slice / integration + Testcontainers).
  - CI/CD, качество кода, конфигурация окружений, секреты.
  - Наблюдаемость: метрики, трейсинг, структурные логи, correlation-id.
  - Локализация/таймзоны как сквозное требование.
  - Модель прав доступа (RBAC + проверка владения ресурсом).
  - Нефункциональные требования (SLA, производительность, лимиты).

================================================================
1. Общие сведения
================================================================

Backend предоставляет REST API для:
  - мобильного приложения (пользователи)
  - веб-админки заведений (владельцы/менеджеры)
  - панели администратора системы (супер-админ)
  - будущих внешних интеграций (POS, партнёры)

Backend отвечает за бизнес-логику, хранение данных, авторизацию,
проверку правил бронирования, уведомления и аналитику.

================================================================
2. Технологии
================================================================

Базовый стек:
  Java 21, Spring Boot 3, Spring Web, Spring Security, Spring Data JPA,
  Hibernate, PostgreSQL, Redis, JWT, MapStruct, Lombok, Gradle,
  OpenAPI (springdoc), Docker, Docker Compose.

Миграции:
  Liquibase (как в tz.txt). Единственный источник изменений схемы —
  changelog'и, схема НЕ создаётся Hibernate (ddl-auto: validate).

Расширения стека (прагматично):
  - Хранилище файлов: S3-совместимое (MinIO локально, S3/облако в prod)
    для аватаров, логотипов, фотографий заведений и меню.
  - Брокер сообщений: RabbitMQ или Kafka — для асинхронной доставки
    уведомлений и интеграционных событий (через transactional outbox).
    На этапе MVP допустимо @Async + БД-outbox, брокер подключается позже
    без изменения контрактов доменных событий.
  - Полнотекстовый / гео-поиск: на MVP — PostgreSQL (pg_trgm + earthdistance
    / PostGIS для радиуса). Elasticsearch/OpenSearch — опционально на этапе 2,
    когда объём и требования к релевантности вырастут.
  - Тестирование: JUnit 5, Testcontainers (PostgreSQL, Redis), Mockito,
    Spring Boot Test, REST-assured / MockMvc, Awaitility (для async).
  - Наблюдаемость: Spring Boot Actuator, Micrometer + Prometheus, Grafana,
    OpenTelemetry (трейсинг), structured JSON-логи (logback).
  - Bucket4j (rate limiting поверх Redis).
  - Resilience4j (retry/timeout/circuit breaker для внешних вызовов:
    Firebase, платежи в будущем, S3).

================================================================
3. Архитектура
================================================================

Модульный монолит (Spring Modulith желательно для контроля границ модулей).

Модули:
  auth, user, restaurant (company/restaurant/hall/table/photo/menu),
  booking, review, favorite, history, notification, search, admin,
  analytics, common (shared: ошибки, пагинация, аудит, файлы, время).

Правила модульности:
  - Межмодульное взаимодействие только через публичные сервисы/интерфейсы
    и доменные события; запрещён доступ к чужим репозиториям/сущностям.
  - У каждого модуля свой пакет, свой набор таблиц, свой префикс API.
  - Общие данные (страны, города, категории кухни) — в common/reference.

Слои внутри модуля:
  controller (DTO in/out) -> service (бизнес-логика, транзакции) ->
  repository (JPA). Маппинг DTO<->Entity — MapStruct.
  Сущности не выходят за пределы service-слоя.

Готовность к выделению в микросервисы:
  - Никаких кросс-модульных foreign key на уровне БД, где модуль потенциально
    отделяемый (связь по id + проверка в коде). В рамках одной БД FK допустимы
    внутри модуля.
  - Доменные события — базис для будущей интеграции через брокер.

================================================================
4. Модель безопасности и права доступа
================================================================

Аутентификация:
  - JWT: короткоживущий Access Token (~15 мин) + Refresh Token (~30 дней).
  - Refresh Token хранится (хэш) в БД/Redis, поддерживается ротация и отзыв
    (logout, logout-all, смена пароля инвалидирует все refresh-токены).
  - Пароли — BCrypt (cost >= 10).
  - Подтверждение email по токену со сроком жизни; восстановление пароля
    по одноразовому токену.
  - OAuth (Google, Apple) — задел на будущее (интерфейс провайдера).

Роли (как в tz.txt): ROLE_USER, ROLE_RESTAURANT_ADMIN,
  ROLE_COMPANY_ADMIN, ROLE_SUPER_ADMIN.

Авторизация (важное дополнение к tz.txt):
  - RBAC + проверка ВЛАДЕНИЯ ресурсом. Недостаточно роли — нужно проверять,
    что менеджер управляет именно СВОИМ заведением/компанией.
    Пример: RESTAURANT_ADMIN может редактировать только рестораны своей
    компании; USER может отменить только своё бронирование.
  - Проверки владения — на уровне сервиса (@PreAuthorize + доменная проверка),
    не только на уровне контроллера.

Прочее:
  - CORS с whitelist доменов из конфигурации.
  - Rate limiting (Bucket4j+Redis): жёсткие лимиты на /auth/*
    (логин, регистрация, восстановление пароля) и на создание бронирований.
  - Валидация входных данных (Jakarta Validation) на всех DTO.
  - Audit log: кто/что/когда изменил ключевые сущности (см. раздел логирование).

================================================================
5. Сквозные контракты API
================================================================

5.1. Формат ошибок — RFC 7807 (Problem Details):
  {
    "type": "https://keepbooking/errors/table-not-available",
    "title": "Table not available",
    "status": 409,
    "detail": "Стол №12 занят на выбранный интервал",
    "instance": "/api/v1/bookings",
    "code": "TABLE_NOT_AVAILABLE",
    "traceId": "abc123",
    "errors": [ {"field": "guestCount", "message": "must be <= capacity"} ]
  }
  Единый @RestControllerAdvice. Коды ошибок стабильны и документированы.

5.2. Пагa­нация/сортировка/фильтрация (единый контракт для всех списков):
  - Query-параметры: page, size (лимит max, напр. 100), sort=field,asc|desc.
  - Ответ-обёртка: { content: [...], page, size, totalElements, totalPages }.
  - Фильтры передаются query-параметрами, описаны в OpenAPI.

5.3. Идемпотентность (дополнение к tz.txt):
  - Для POST, создающих деньги/бронь/внешние эффекты — заголовок
    Idempotency-Key. Повторный запрос с тем же ключом возвращает исходный
    результат, а не создаёт дубль. Ключи хранятся в Redis/БД с TTL.

5.4. Версионирование:
  - Префикс /api/v1. Ломающие изменения — только в новой мажорной версии.
  - Аддитивные изменения (новые поля) не считаются ломающими.

5.5. Время и локализация (сквозное):
  - Все временные метки хранятся и передаются в UTC (Instant / timestamptz).
  - Клиент передаёт/получает таймзону; бизнес-правила бронирования
    (рабочие часы, «дата прошла») считаются в таймзоне ЗАВЕДЕНИЯ.
  - Языковые тексты (уведомления, статусы) — через message bundles / Accept-Language.

================================================================
10. Доступность столиков и слоты (НОВЫЙ КЛЮЧЕВОЙ РАЗДЕЛ)
================================================================

Это главный пробел исходного tz.txt: не описано, КАК считается доступность.

10.1. Параметры бронирования заведения (настраиваются рестораном):
  - slotDuration (шаг сетки, напр. 30 мин) или свободный интервал.
  - defaultBookingDuration (напр. 2 часа) — сколько стол занят по умолчанию.
  - bufferBetweenBookings (уборка/подготовка, напр. 15 мин).
  - minAdvance / maxAdvance (за сколько можно/нельзя бронировать: напр.
    минимум за 30 мин, максимум на 60 дней вперёд).
  - autoConfirm (авто-подтверждение или ручное менеджером).
  - holdTimeout (сколько держится PENDING до авто-отмены).

10.2. Эндпоинт доступности:
  GET /api/v1/restaurants/{id}/availability?date=&from=&to=&guests=
  Возвращает для каждого столика (или слота) статус:
  свободен / занят / частично, с учётом рабочих часов, буферов и существующих
  броней. Именно этот эндпоинт питает выбор столика в UI.

10.3. Правило пересечения интервалов:
  Стол занят, если запрашиваемый [timeFrom, timeTo) пересекается с любым
  активным бронированием (PENDING/CONFIRMED) того же стола с учётом буфера.
  Формула пересечения: newFrom < existingTo+buffer AND newTo+buffer > existingFrom.

================================================================
11. Бронирование
================================================================

Поля: id, restaurantId, tableId, userId, date, timeFrom, timeTo (UTC),
  guestCount, comment, status, source (USER/ADMIN), createdBy,
  confirmedBy, cancelledBy, cancelReason, idempotencyKey,
  createdAt, updatedAt.

Статусы (как в tz.txt): PENDING, CONFIRMED, REJECTED, CANCELLED,
  COMPLETED, NO_SHOW.
Разрешённые переходы описать явно (state machine), запрещённые — 409/422.
  PENDING -> CONFIRMED/REJECTED/CANCELLED
  CONFIRMED -> COMPLETED/CANCELLED/NO_SHOW
  Терминальные статусы неизменяемы.

11.1. Проверки перед созданием (расширяют tz.txt):
  - столик существует, активен и принадлежит указанному заведению;
  - интервал попадает в рабочие часы заведения (в его таймзоне);
  - дата/время не в прошлом; в пределах min/maxAdvance;
  - guestCount <= capacity столика (и >= minCapacity, если задано);
  - столик свободен на интервал (правило 10.3);
  - у пользователя нет конфликтующей активной брони на то же время;
  - лимит на число активных броней пользователя (анти-абьюз).

11.2. Конкурентность / защита от double-booking (НОВОЕ, критично):
  Два пользователя не должны забронировать один стол на один интервал.
  Решение (одно из, задокументировать выбор):
  - Уникальный частичный индекс / exclusion constraint в PostgreSQL
    (btree_gist по tableId + tsrange интервала для активных статусов) —
    предпочтительно, гарантия на уровне БД; ИЛИ
  - Пессимистичная блокировка стола (SELECT ... FOR UPDATE) в транзакции; ИЛИ
  - Оптимистичная блокировка (@Version) с ретраем.
  При конфликте — 409 TABLE_NOT_AVAILABLE (идемпотентно к Idempotency-Key).

11.3. Жизненный цикл PENDING:
  Если autoConfirm выключен, бронь висит PENDING до подтверждения менеджером
  или до holdTimeout, после чего авто-отмена (scheduled job).
  Напоминания и авто-перевод в NO_SHOW/COMPLETED — по расписанию.

11.4. Отмена/перенос: правила и дедлайны отмены пользователем;
  уведомления обеим сторонам; освобождение слота триггерит событие
  «стол освободился» (для листа ожидания в будущем).

================================================================
12. Поиск и карта
================================================================

Поиск по: названию (pg_trgm), городу, кухне (m2m), рейтингу,
  среднему чеку, радиусу от точки (гео), НАЛИЧИЮ свободных столиков
  на дату/время/кол-во гостей (интеграция с разделом 10).
  Пагинация + сортировка (рейтинг, расстояние, популярность).

Карта: GET рестораны в bbox/радиусе с координатами, статусом, рейтингом,
  признаком «есть свободные столы сейчас». Кэш в Redis с коротким TTL.

Этап 2: перенос поиска в Elasticsearch/OpenSearch при росте нагрузки.

================================================================
13. Отзывы, избранное, история
================================================================

Отзывы: id, restaurantId, userId, bookingId, rating (1..5), text,
  status (PUBLISHED/HIDDEN), createdAt, updatedAt.
  Правило: отзыв можно оставить только после брони со статусом COMPLETED,
  и не более одного отзыва на бронирование. Пересчёт rating/reviewsCount
  заведения (денормализация) — при публикации/сокрытии/удалении отзыва.

Избранное: (userId, restaurantId) уникальная пара; add/remove/list.

История: посещённые рестораны, история бронирований, история поисков
  (для рекомендаций в будущем). Хранить события через тот же outbox/лог,
  не блокируя основной поток.

================================================================
14. Уведомления и Push
================================================================

Уведомления (in-app): id, userId, title, message, type, read,
  createdAt, dataPayload(json, deep-link).
  Эндпоинты: список (пагинация), пометить прочитанным, счётчик непрочитанных.

Push (Firebase FCM):
  Типы: подтверждение брони, отмена, напоминание, «стол освободился»,
  изменение статуса. Устройства пользователя: таблица push-токенов
  (userId, token, platform, updatedAt), очистка невалидных токенов.

Доставка асинхронно (transactional outbox):
  1) в одной транзакции с бизнес-операцией пишем событие в outbox;
  2) отдельный воркер публикует в брокер / шлёт push с ретраями (Resilience4j);
  3) идемпотентность доставки, dead-letter для неудачных.
  Настройки пользователя: включение/выключение каналов и типов уведомлений.

================================================================
15. Аналитика (для заведения и системы)
================================================================

Для заведения: число броней, отмен, NO_SHOW, конверсия PENDING->CONFIRMED,
  популярные часы/дни, популярные столики, число уникальных гостей,
  средняя загрузка, средний чек (если ведётся), динамика рейтинга.
  Отдавать за период (from/to), с группировкой (день/неделя/месяц).
  Читать из отдельных read-моделей/материализованных представлений,
  чтобы не грузить транзакционные таблицы.

Для системы (супер-админ): пользователи (рост, активные), компании,
  заведения по статусам, суммарные брони/отмены, топ-города/кухни.

================================================================
16. Администрирование
================================================================

Компания/менеджер: CRUD ресторанов (в рамках своей компании), залы, столики,
  редактирование схемы, меню, фото, управление бронированиями (подтвердить/

Супер-админ: пользователи (просмотр, блок/разблок), компании (просмотр, блок),

  управление справочниками (страны, города, кухни), системная статистика,
  просмотр audit-log.

Все админ-действия защищены проверкой роли И владения ресурсом.

================================================================
17. Наблюдаемость, логирование, аудит
================================================================

Логи: структурные JSON, уровень настраивается; в каждый лог — correlationId
  (из заголовка X-Request-Id или сгенерированный), userId (если есть).
  Без PII/секретов/токенов/паролей в логах.

Аудит (отдельная таблица audit_log): actorId, action, entityType, entityId,
  before/after (важные поля), ip, timestamp. Пишется для: авторизации,
  CRUD ресторанов/залов/столиков, создания/отмены брони, модерации,
  изменения ролей, блокировок.

Метрики: Micrometer -> Prometheus (HTTP latency, error rate, брони по статусам,
  доставка уведомлений, использование кэша, пул БД). Дашборды в Grafana.
Трейсинг: OpenTelemetry, сквозной traceId в ответах об ошибках.
Health: Actuator /health (liveness/readiness), /info, /metrics
  (закрыты авторизацией/сетевыми правилами).

================================================================
18. Хранение файлов
================================================================

S3-совместимое хранилище (MinIO локально). Загрузка через presigned URL
  либо через backend с валидацией (тип, размер, антивирус-хук опц.).
  В БД — только ключ объекта и метаданные. Генерация превью/ресайз —
  асинхронно. Приватные и публичные бакеты по назначению.

================================================================
19. Кэширование (Redis)
================================================================

Кэшировать: справочники (страны/города/кухни, длинный TTL),
  карточки заведений (средний TTL, инвалидация при изменении),
  результаты карты/поиска (короткий TTL), rate-limit счётчики,
  idempotency-ключи, refresh-token whitelist/blacklist, email-верификация.
  Явно задокументировать ключи и TTL; предусмотреть инвалидацию.

================================================================
20. Конфигурация, окружения, секреты
================================================================

Профили Spring: local, dev, test, prod. Конфиг через переменные окружения
  (12-factor). Секреты не в git (env / vault / docker secrets).
  Docker Compose поднимает всё локально: app, postgres, redis, minio,
  (rabbitmq), prometheus, grafana. Отдельный compose для тестов не требуется
  — Testcontainers.

================================================================
21. Тестирование (НОВОЕ, требование к качеству)
================================================================

Уровни:
  - Unit: бизнес-правила (проверки брони, переходы статусов, доступность).
  - Slice: @WebMvcTest (контроллеры/валидация), @DataJpaTest (репозитории/запросы).
  - Integration: Testcontainers (реальные PostgreSQL/Redis), сценарии end-to-end
    на уровне API (MockMvc/REST-assured), в т.ч. конкурентное бронирование
    (параллельные запросы на один стол -> ровно одна успешная бронь).
Обязательное покрытие критичных путей: авторизация, создание/отмена брони,
  проверка доступности, права доступа (owner checks).
Тестовые данные — фабрики/фикстуры; БД чистая между тестами.
Цель по покрытию доменной логики ориентировочно >= 70%.

================================================================
22. CI/CD и качество кода
================================================================

CI (GitHub Actions/GitLab CI): сборка Gradle, запуск тестов (Testcontainers),
  статический анализ (Checkstyle/Spotless форматирование, PMD/SpotBugs),
  проверка уязвимостей зависимостей (OWASP Dependency-Check),
  сборка Docker-образа, публикация в registry.
Ветвление: PR обязателен, зелёный CI как условие мержа.
Миграции Liquibase проверяются в CI (валидность changelog, отсутствие конфликтов).

================================================================
23. Нефункциональные требования
================================================================

  - Производительность: p95 чтения < 300 мс на типовых запросах при кэше.
  - Доступность цель: 99.5% (MVP), graceful shutdown, readiness-проба.
  - Пагинация обязательна для всех коллекций (нет unbounded-выборок).
  - Ограничения размера запросов/загрузок, таймауты внешних вызовов.
  - Обратная совместимость API в пределах мажорной версии.
  - Идемпотентность и повторяемость операций записи.
  - Безопасность: OWASP Top-10 (инъекции, IDOR/owner-checks, авторизация,
    rate limiting, валидация, безопасные заголовки).

25. План поэтапной реализации (уточнён)
================================================================

Этап 1 (MVP, «рабочий скелет production-уровня»):
  - Авторизация, роли, refresh/verify/reset, owner-checks.
  - Справочники (страны/города/кухни).
  - Компании, рестораны, рабочие часы, залы, столики (схема с координатами).
  - ДОСТУПНОСТЬ столиков + бронирование с защитой от double-booking
    и идемпотентностью (ядро продукта — делать сразу правильно).
  - Личный кабинет, базовая админка ресторана, модерация супер-админом.
  - Единый формат ошибок, пагинация, валидация, Swagger.
  - Liquibase, Docker Compose (app/postgres/redis/minio), Testcontainers,
    базовый CI, Actuator health.

Этап 2:
  - Избранное, отзывы (с привязкой к COMPLETED), история.
  - Поиск с фильтрами, карта, кэширование.
  - Push-уведомления (Firebase) через outbox/брокер, in-app уведомления.
  - Метрики/трейсинг/дашборды, rate limiting, audit-log расширенный.

Этап 3:
  - Аналитика (read-модели), полноценный мониторинг (Prometheus/Grafana).
  - Elasticsearch для поиска (при необходимости).
  - Лист ожидания (событие «стол освободился» уже есть на этапе 1).
  - Программа лояльности, предзаказ блюд, онлайн-оплата депозитов,
    интеграции с POS, OAuth (Google/Apple), мультиязычность.

Принцип: архитектура поддерживает все модули с самого начала (границы модулей,
доменные события, outbox), но реализуются они поэтапно — быстрый MVP без
последующих серьёзных переделок.
